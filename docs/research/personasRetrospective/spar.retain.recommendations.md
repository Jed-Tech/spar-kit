# Recommendations for `spar.retain`

## Goal

Keep `spar.retain` clean and durable while making the closeout feel like a natural end to the work rather than an administrative afterthought.

## What the retrospective surfaced

`spar.retain` worked well in this session. The main opportunity is not structural correctness but user-facing clarity around why retention matters.

## Main recommendation

Make only small clarity improvements.

## Recommended changes

### 1. Reinforce the purpose of retention in user-facing phrasing

The skill already treats retained specs as the source of truth. It would help to make that value proposition slightly more visible in closeout language so users understand why this step exists.

### 2. Encourage concise note of broader doc implications

When no broader documentation changes are needed, the skill should say so briefly and confidently. That keeps the flow crisp and reassures the user that this step was considered, not skipped.

### 3. Keep archive behavior strict

The archive rule is good and should remain strict. That is one of the strongest parts of the skill.

## What not to change

- Do not weaken the approval gate for broader repo docs
- Do not soften the archive rule
- Do not let retention turn into a long retrospective by default

## Recommended priority

Low. This skill is in comparatively good shape.
